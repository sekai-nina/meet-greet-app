# 要件定義: 日向坂46 ミーグリ参加記録 & 購買計画アプリ

## 1. プロダクト概要

日向坂46 のミート&グリート (ミーグリ) の **参加記録** と **購買計画** を管理するスマホアプリ。
自分の記録を残すだけでなく、他ユーザーの記録を集計して各日程の売れ行き (完売状況) を可視化する。

### 1.1 ジョブ (JTBD)

ユーザーが達成したいジョブを定義する。1人のユーザーが複数のジョブを同時に持ちうる。

#### J1: 推しとたくさん話したい

- **状況**: 新シングルのミーグリが発表された
- **現状のペイン**: 購買計画がスプレッドシートや手書きで煩雑 / 当落は forTUNE music のサイトやメールを都度確認 / 初回限定盤のシリアルは何枚どこに投げたか把握しづらい
- **期待する結果**: できるだけ多くの枠を確保できている
- **アプリでの解決**: 推しメンを設定すると計画・提案の軸になる (F3) / 確保状況と完売状況を見ながら受付次ごとに計画 (F6) / 理想マトリックスで目標との差分を管理 (F7, オプション) / 受付次ごとの申込・当落を記録 (F4) / 枚数ベースでシリアル投入先を一元管理 (F8) / ポリシーに基づく投げ先提案 (F9)

#### J2: 推しの完売を増やしたい

- **状況**: 推しの完売数を伸ばしたい / 未完売枠がある
- **現状のペイン**: 完売情報が複数サイト・SNS に散在 / 投げ先の判断が属人的で SNS の影響力ある人が先導している
- **期待する結果**: 自分の応募が完売達成につながる
- **アプリでの解決**: ユーザー投稿の集計で完売状況を一元化 (F5, F11) / 購買ポリシーに基づきアプリが投げ先を提案 (F9)

#### J3: 初めてのミーグリに迷わず参加したい

- **状況**: ミーグリに興味はあるが仕組みがわからない
- **現状のペイン**: SNS や口コミで手探り。どこに何枚投げればいいか判断できない
- **期待する結果**: 何をどう投げればいいか提案してもらえる
- **アプリでの解決**: ポリシー提案で迷わず投げられる (F9) / 完売状況や人気枠の傾向を参考にできる (F5)

#### J4: 多くのメンバーに会いたい

- **状況**: 複数メンバーの枠に投げたい
- **現状のペイン**: メンバーごとにバラバラに管理しており横断的な把握が難しい
- **期待する結果**: 被りを管理しつつ効率よく回れる
- **アプリでの解決**: 受付次別計画で横断ビューから被りや全体像を把握 (F6) / 被り上限を考慮した提案 (F9)

#### J5: 特定の日程で参加したい

- **状況**: 推しの誕生日に絶対行きたい / この日は行けない
- **現状のペイン**: 希望日程に絞った計画を手動で管理する必要がある
- **期待する結果**: 希望日程に集中した計画ができている
- **アプリでの解決**: 参加可能/希望日程を設定すると、計画がその日程に絞られる (F6) / 最終受付の自動判定 (F6) / 日程に基づく提案 (F9)

#### J6: 参加記録を残して振り返りたい

- **状況**: ミーグリに参加した / 過去のシングルを振り返りたい
- **現状のペイン**: 記録用の別アプリを使う必要があり、購買計画や当落管理と分散してしまう
- **期待する結果**: 体験が記録として残り、いつでも振り返れる
- **アプリでの解決**: 構造化された記録として蓄積・一覧表示 (F4) / シングルごとの振り返りメモ (F12)

#### J7: ミーグリの受付・参加をスムーズにこなしたい

- **状況**: 受付期間中 / ミーグリ当日
- **現状のペイン**: 受付スケジュールは公式サイトを都度確認 / 受付・支払い・当日の期限を自分でカレンダー管理していて忘れがち
- **期待する結果**: 必要なアクションを適切なタイミングで漏れなくこなせる
- **アプリでの解決**: アプリ内で受付次・最終受付を一覧表示 (F2) / プッシュ通知でリマインド (F10) / 当選した部をカレンダーアプリに同期 (F10)

**制約パラメータ** (全ジョブ共通):
- 予算・枚数上限
- 参加可能/希望日程
- 被り上限 (同一部で複数メンバー)

## 2. 機能要件

### 2.1 v1 (MVP) — 記録と可視化

#### F1: 認証

- Supabase Auth によるログイン
- マジックリンク (メール) または匿名サインイン
- 匿名→メール連携へのアップグレード対応

#### F2: イベント情報管理

- リリース情報の登録 (シングル / アルバム)
  - タイトル / 発売日 / センター / リリース種別
- イベント情報の登録 (1 リリースに複数イベントが紐づく)
  - イベント形式: 通常版オンラインミーグリ / 初回限定盤オンラインミーグリ / リアルミーグリ / サイン会
  - 開催日一覧 (DAY1〜DAYn)
  - 各開催日の部数と時間割 (形式によって異なる)
  - 各開催日の参加メンバー (基本全員出演。欠席 / 卒業メンバー専用日程の追加など例外に対応)
  - サイン会: 開催日・対象メンバー・当選枠数 (各メンバー 20 名)
  - 受付次スケジュール (第1次〜第N次の日時・申込回数上限)
- 一覧表示・検索
  - リリース種別 (シングル/アルバム) / イベント形式でフィルタ
  - シングル名・アルバム名で検索
  - 開催日が近い順にソート
- **初期データは管理者 or コミュニティ投稿で登録** (公式 API は存在しない)

#### F3: 推しメン設定

- ユーザーごとに推しメン (1 人 or 複数) を設定

#### F4: 参加記録

参加状況のスナップショットと、受付次ごとの申込・当落履歴を一元管理する。

- **参加状況**: イベント × 開催日 × 部 × メンバー ごとに現在のステータスを表示・管理
  - ステータス: 未申込 / 申込済 / 当選 / 落選 / 参加済 / キャンセル
  - メモ欄 (自由記述)
  - 入力 UI: マトリックス形式 (行=開催日, 列=部) でタップ入力
  - サイン会の記録にも対応
- **受付次別の申込・当落**: 受付次 × 開催日 × 部 × メンバー ごとに申込枚数・当選枚数を記録
  - 当落結果の登録時に参加状況のステータスを自動更新
  - 投げた枚数 > 当選枚数の場合、その枠の完売推定に活用 (F5 と連携)
- 一覧・集計
  - シングル別に参加記録を一覧表示
  - フィルタ: メンバー / 開催日 / ステータス
  - 集計: 合計参加部数 / 合計金額
  - 受付次ごとの申込履歴を一覧表示

#### F5: 完売状況の集約表示

- 他ユーザーの参加記録 (F4) を集計
- メンバー × 開催日 × 部 の完売率 (当選報告数ベース) を表示
- 一部当選データから完売を推定 (複数ユーザーの報告で信憑性を判断)
- 未完売枠のフィルタ (J2 向け)
- 人気枠のソート (当選報告数が多い順)
- **注意**: ユーザー投稿ベースのため、あくまで参考値であることを明示

### 2.2 v2 — 購買計画

#### F6: 受付次別計画

v2 の中核機能。完売状況 (F5) と確保状況 (F4 の当落データ) を見ながら、次の受付次でどこに何枚投げるかを計画する。

- **確保状況ビュー**: F4 の当落記録から、開催日×部ごとの確保済み枠数を自動集計・表示
- **受付次ごとの計画マトリックス**: その次で何部投げるかを入力 (行=開催日, 列=部)
- 完売状況 (F5) を参照しながら投げ先を判断
- 応募対象日程の自動判定 (最終受付ルール: 開催 8〜11 日前)
- 応募不可の開催日をグレーアウト表示
- 参加不可/希望日程の設定に対応 (J5 向け)
- 複数メンバーの計画を横断表示 (J4 向け)
- 通常盤・初回限定盤それぞれの計画に対応

#### F7: 理想マトリックス (オプション)

事前に目標を立てて計画的に進めたいユーザー向け。

- **理想マトリックス**: 行=開催日, 列=部, セル=目標部数
- **差分表示**: 理想 − 確保 = 残り必要数 を自動計算・色分け表示
- 設定しなくても F6 の受付次別計画は利用可能

#### F8: 予算・枚数管理

- **通常盤**: 単価 1,200 円/枚、送料 700 円/注文で金額を自動計算
- **初回限定盤**: 枚数 (シリアル数) の管理を主軸とする。金額は参考値 (購入先で単価が変動するため)
- 受付次ごと・累計の投入枚数可視化
- 通常盤と初回限定盤を分けて管理

#### F9: 購買ポリシーに基づく計画提案

ユーザーが選択したポリシーとパラメータに基づき、次の受付次でどこに何枚投げるかを提案する。

**ポリシー** (ユーザーが選択):

| ポリシー | 提案の方向性 |
|---------|------------|
| 完売優先 | 未完売で完売に近い枠を優先提案 |
| 推し最大化 | 推しメンの全枠にできるだけ均等に |
| 当選確率重視 | 倍率が低そうな枠を優先提案 |
| 日程分散 | できるだけ多くの日程に分散 |
| 日程集中 | 指定した日程に集中投入 |

**パラメータ** (ユーザーが設定):

- 参加可能日程
- 被り上限 (被りなし / n人まで / 制限なし) — 多推し時の同一部での複数メンバー当選制御
- 予算・枚数上限

### 2.3 v3 — コミュニティ・通知

#### F10: 通知・リマインド

- 受付開始・締切のリマインダー
- 支払い期限のリマインダー
- 当日の参加リマインド (「あなたのチケットの第n部があとX分後に始まります」)
- 完売速報 (推しメンの枠が完売したとき)
- 当選した部のカレンダーアプリ同期 (iOS カレンダー / Google カレンダー)

#### F11: コミュニティ完売表

- メンバー全員の完売状況を表形式で表示
- シングルごとの完売推移グラフ
- ユーザー報告数に基づく信頼度スコア

#### F12: 振り返り・メモ

- シングルごとの振り返りメモ
- 「次回への学び」欄

## 3. 非機能要件

### 3.1 パフォーマンス

- 一覧画面の初期表示: 2 秒以内
- マトリックス操作 (タップ→反映): 100ms 以内

### 3.2 セキュリティ

- Supabase RLS により、自分の記録は自分のみ編集可能
- 集計データは読み取り専用で全ユーザーに公開
- 個人を特定できる情報は集計結果に含めない
- **クライアントに含まれる環境変数 (`EXPO_PUBLIC_*`) は公開情報として扱う** — Supabase anon key は RLS で保護されている前提で公開可。service_role key は絶対にクライアントに含めない
- 書き込み系の操作は将来的に Supabase Edge Functions 経由に移行し、サーバー側で検証・監査ログを残すことを検討する
- 重複送信・多重タップ対策として、書き込み API には idempotency key を導入する (v1 実装時)

### 3.2.1 OTA 互換性管理

- EAS Update (OTA) でスキーマ変更を伴う更新を配信する場合、旧クライアントが壊れるリスクがある
- DB スキーマに `schema_version` を持たせ、アプリ起動時に互換性を確認する仕組みを検討する
- `runtimeVersion` の `appVersion` ポリシーで OTA の適用範囲を制御する

### 3.2.2 タイムゾーン

- ミーグリの受付締切・開催日は **JST (Asia/Tokyo)** 基準
- DB には `timestamptz` (UTC) で保存し、表示時に JST に変換する
- 受付締切の境界 (例: 14:00 JST) を扱う際は UTC 変換のズレに注意する

### 3.3 対応プラットフォーム

- iOS 15+
- Android 12+
- (Web は v1 スコープ外、将来対応検討)

### 3.4 多言語対応

- v1: 日本語のみ
- 将来: 英語対応検討

### 3.5 機密情報管理

- **平文の `.env` ファイルは絶対にリポジトリにコミットしない**
- 環境変数の管理には **dotenvx-rs** (Rust 製 dotenvx CLI) を使用する
  - `dotenvx encrypt` で `.env` を暗号化し、暗号化済みファイルをコミットする
  - 復号鍵は `$HOME/.dotenvx/.env.keys.json` にグローバル管理される (プロジェクト内に `.env.keys` を置かない)
  - 開発者が増えた場合、復号鍵 (`DOTENV_PRIVATE_KEY`) は口頭・DM 等でリポジトリ外で共有する
  - 実行時は `dotenvx run -- npx expo start` で復号・注入する
  - 本番環境では `DOTENV_PRIVATE_KEY` を CI/CD のシークレットに設定する
- `.gitignore` に以下を必ず含める:
  - `.env.keys` (万が一ローカルに生成された場合の安全策)
  - `.env` (平文の環境変数)
- **機密情報の漏洩防止は 2 層で対策する**:
  - **ローカル (pre-commit)**: Gitleaks でコミット前に機密情報パターンをスキャン・ブロック
  - **リモート (GitHub Push Protection)**: push 時に既知のシークレットパターンを検出・ブロック (public リポジトリではデフォルト有効)
- 誤検知は `.gitleaksignore` に fingerprint を追加して除外する
- 新規開発者は `make setup` で自動的に pre-commit フックが設定される

### 3.6 インフラ・運用

- バックエンドは **Supabase のみ** で構成 (別途 API サーバーは立てない)
  - Auth / Postgres (RLS) / Edge Functions で全機能をカバー
- 初期は **Free プラン** で運用し、ユーザー定着後に Pro ($25/月) へ移行
- Free プランの自動停止対策 (7 日間無アクセスでプロジェクトが pause される)
  - GitHub Actions 等の cron で 1 日 1 回 Supabase へ軽量クエリを実行し、停止を回避する

### 3.7 環境構成

**dev** と **prd** の 2 環境で運用する。stg は当面不要 (チーム拡大時に必要に応じて追加)。

| 環境 | 用途 | Supabase プロジェクト | 環境変数ファイル |
|------|------|----------------------|-----------------|
| dev | ローカル開発・動作確認 | `supabase start` (ローカル) または Free プランの開発用プロジェクト | `.env` |
| prd | 本番 | Supabase プロジェクト (Free → Pro) | `.env.production` |

- 環境ごとに Supabase プロジェクトを分離し、本番データと開発データを混在させない
- 環境変数ファイルはすべて dotenvx-rs で暗号化管理する
- マイグレーションは `supabase/migrations/` で一元管理し、dev・prd 両方に同じ SQL を適用する

## 4. データモデル (概要)

```
releases  (シングル or アルバム)
  id, title, number, release_date, center_member,
  release_type (single|album)

events  (1リリースに複数紐づく)
  id, release_id,
  format (regular_online|limited_online|real|autograph)

members
  id, name, generation, birthday

event_days
  id, event_id, day_number, date, slot_count

event_day_members  (出演メンバー)
  event_day_id, member_id

reception_rounds  (受付次)
  id, event_id, round_number, start_at, end_at,
  max_applications (申込回数上限: 1次=1, 2次=3, 3次以降=5)

reception_round_targets  (受付次 × 応募可能な開催日)
  reception_round_id, event_day_id, is_final_round

user_oshi_members  (推しメン)
  user_id, member_id

participation_records  (参加記録 — 各枠の現在の状態)
  id, user_id, event_id, event_day_id, slot_number, member_id,
  status (notApplied|applied|won|lost|attended|cancelled), memo

round_applications  (受付次別の申込・当落)
  id, user_id, reception_round_id, event_day_id, slot_number, member_id,
  applied_count, won_count

purchase_plans  (購買計画)
  id, user_id, event_id

plan_ideal_slots  (理想マトリックス)
  plan_id, event_day_id, slot_number, target_count

plan_actual_slots  (確保状況マトリックス)
  plan_id, event_day_id, slot_number, secured_count

plan_round_slots  (受付次別計画)
  plan_id, reception_round_id, event_day_id, slot_number, planned_count

user_purchase_policies  (購買ポリシー設定)
  user_id, event_id,
  policy (soldout_priority|oshi_maximize|win_rate|date_spread|date_focus),
  available_days (参加可能日程),
  overlap_limit (被り上限),
  budget_limit, count_limit (予算・応募枚数上限)
```

## 5. 画面構成

| 画面 | 概要 | 主な操作 |
|------|------|----------|
| ログイン | マジックリンク or 匿名 | メール入力 / 匿名ログイン |
| ホーム | 直近のイベント・自分の記録サマリー | イベント選択 |
| イベント一覧 | リリース別イベント一覧 | 検索 / フィルタ |
| イベント詳細 | 開催日・受付次・完売状況 | 開催日選択 |
| 参加記録 | マトリックス形式で記録入力・一覧表示 | セルタップ→ステータス切替 / 受付次別の当落入力 / フィルタ / 集計 |
| 完売状況 | メンバー×日程×部の完売率表示 | フィルタ / ソート |
| 購買計画 (v2) | 理想/確保/受付次別マトリックス | セル入力 / 差分確認 |
| 計画提案 (v2) | ポリシー設定・提案結果表示 | ポリシー選択 / パラメータ設定 / 提案の採用 |
| 設定 | 推しメン設定 / アカウント | 推しメン追加・削除 |
| カタログ (dev) | コンポーネント一覧 (__DEV__ のみ) | バリエーション確認 / 外部共有 |

## 6. 技術スタック

CLAUDE.md に記載の通り:

- **フロントエンド**: React Native + Expo (TypeScript)
