# 要件定義: 日向坂46 ミーグリ参加記録 & 購買計画アプリ

## 1. プロダクト概要

日向坂46 のミート&グリート (ミーグリ) の **参加記録** と **購買計画** を管理するスマホアプリ。
自分の記録を残すだけでなく、他ユーザーの記録を集計して各日程の売れ行き (完売状況) を可視化する。

### 1.1 ペルソナ

ペルソナは排他的な分類ではなく **行動特性の軸** として定義する。1人のユーザーが複数の特性を持ちうる。

| ID | 名前 | 特徴 | 主な関心 |
|----|------|------|---------|
| P1 | ヘビー推し型 | 1〜2 人の推しメンに大量に投げる。とにかく推しとたくさん話したい | 確保数 vs 目標数、予算・枚数管理、受付スケジュール |
| P2 | 完売貢献型 | 推しメンの完売に貢献したい。完売数=人気指標なので推しの「数字」を伸ばしたい | 完売状況のリアルタイム把握、未完売枠の可視化、投げ先の集約 |
| P3 | ビギナー型 | ミーグリ未経験 or 数回程度。仕組みや投げ方がよくわからない | イベント構造の理解、提案機能の恩恵を最も受けやすい層 |
| P4 | 多推し型 (箱推し) | 多くのメンバーと話したい。1 日で複数メンバーを回る | 複数メンバーの横断管理、被り制御 |
| P5 | 日程希望型 | 特定の日程に強い希望がある (推しの誕生日に絶対行きたい、仕事で行けない日がある等) | 参加可能/優先日程の設定、最終受付の把握 |
| P6 | 予算重視型 | 使える金額・枚数に上限がある。少ない枚数で最大限の結果を得たい | 枚数管理、受付次ごとの投入枚数可視化 |

### 1.2 解決する課題

| 課題 | 現状 | アプリで解決 |
|------|------|-------------|
| 購買計画が煩雑 | スプレッドシートや手書きで管理 | マトリックス UI で直感的に計画 |
| 当落管理が面倒 | forTUNE music のサイトやメールを都度確認 | アプリでさっと登録・確認でき、残り必要数も自動計算 |
| 完売情報が散在 | 複数サイト・SNS を巡回 | ユーザー投稿の集計で完売状況を一元化 |
| 参加記録が残らない | 記憶頼み or メモアプリ | 構造化された記録として蓄積 |
| 受付スケジュール把握 | 公式サイトを都度確認 | アプリ内で受付次・最終受付を一覧表示 |
| 受付・支払い・当日の期限を忘れる | 自分でカレンダー管理 | プッシュ通知でリマインド |
| 初心者がどう投げればいいかわからない | SNS や口コミで手探り | ポリシー提案で迷わず投げられる |
| 投げ先の判断が属人的 | SNS で影響力のある人が先導 | 購買ポリシーに基づきアプリが投げ先を提案 |
| 初回限定盤のシリアル管理が煩雑 | 何枚どこに投げたか把握しづらい | 枚数ベースで投入先を一元管理 |
| 複数推しの計画が横断しにくい | メンバーごとにバラバラに管理 | 横断ビューで被りや全体像を把握 |

## 2. 機能要件

### 2.1 v1 (MVP)

#### F1: 認証

- Supabase Auth によるログイン
- マジックリンク (メール) または匿名サインイン
- 匿名→メール連携へのアップグレード対応

#### F2: リリース・イベント管理

- リリース情報の登録 (シングル / アルバム)
  - タイトル / 発売日 / センター / リリース種別
- イベント情報の登録 (1 リリースに複数イベントが紐づく)
  - イベント形式: 通常版オンラインミーグリ / 初回限定盤オンラインミーグリ / リアルミーグリ / サイン会
  - 開催日一覧 (DAY1〜DAYn)
  - 各開催日の部数と時間割 (形式によって異なる)
  - 各開催日の参加メンバー
  - サイン会: 開催日・対象メンバー・当選枠数 (各メンバー 20 名)
- **初期データは管理者 or コミュニティ投稿で登録** (公式 API は存在しない)

#### F3: 推しメン設定

- ユーザーごとに推しメン (1 人 or 複数) を設定
- 推しメンの出演日をシングル情報から自動抽出・ハイライト表示

#### F4: 参加記録の入力

- イベント × 開催日 × 部 × メンバー を選択して記録
- ステータス: 当選 / 落選 / 未申込 / キャンセル
- メモ欄 (自由記述)
- 入力 UI: マトリックス形式 (行=開催日, 列=部) でタップ入力
- サイン会の記録にも対応

#### F5: 自分の記録一覧

- シングル別に参加記録を一覧表示
- フィルタ: メンバー / 開催日 / ステータス
- 集計: 合計参加部数 / 合計金額

#### F6: イベント一覧・検索

- 登録済みリリース・イベントの一覧
- リリース種別 (シングル/アルバム) でフィルタ
- イベント形式 (通常版オンライン/初回限定盤オンライン/リアル/サイン会) でフィルタ
- シングル名・アルバム名で検索
- 開催日が近い順にソート

#### F7: 完売状況の集約表示

- 他ユーザーの参加記録を集計
- メンバー × 開催日 × 部 の完売率 (当選報告数ベース) を表示
- 未完売枠のフィルタ (完売貢献型ペルソナ向け)
- 人気枠のソート (当選報告数が多い順)
- **注意**: ユーザー投稿ベースのため、あくまで参考値であることを明示

### 2.2 v2 (購買計画機能)

#### F8: 購買計画マトリックス

- **理想マトリックス**: 行=開催日, 列=部, セル=目標部数
- **確保状況マトリックス**: 同じ形式, セル=確保済み部数
- **差分表示**: 理想 − 確保 = 残り必要数 を自動計算・色分け表示
- 推しメンの出演日のみ表示 or 全日程表示の切り替え
- 参加不可日程のグレーアウト / 除外 (日程制約型ペルソナ向け)
- 複数メンバーの計画を横断表示 (多推し型ペルソナ向け)
- 通常盤・初回限定盤それぞれの計画に対応

#### F9: 受付次別計画

- 受付スケジュール管理 (第1次〜第N次の日時)
- 受付次ごとの計画マトリックス (その次で何部投げるか)
- 応募対象日程の自動判定 (最終受付ルール: 開催 8〜11 日前)
- 応募不可の開催日をグレーアウト表示

#### F10: 予算・枚数管理

- **通常盤**: 単価 1,200 円/枚、送料 700 円/注文で金額を自動計算
- **初回限定盤**: 枚数 (シリアル数) の管理を主軸とする。金額は参考値 (購入先で単価が変動するため)
- 受付次ごと・累計の投入枚数可視化
- 通常盤と初回限定盤を分けて管理

#### F11: 受付次の当落結果入力 & 計画更新

- 当落結果を一括入力
- 確保状況マトリックスの自動更新
- 次の受付次への推奨投入数を提案 (残り必要数ベース)

#### F15: 購買ポリシーに基づく計画提案

ユーザーが選択したポリシーとパラメータに基づき、次の受付次でどこに何枚投げるかを提案する。

**ポリシー** (ユーザーが選択):

| ポリシー | 提案の方向性 |
|---------|------------|
| 完売優先 | 未完売で完売に近い枠を優先提案 |
| 推し最大化 | 推しメンの全枠にできるだけ均等に |
| 当選確率重視 | 倍率が低そうな枠を優先提案 |
| 日程分散 | できるだけ多くの日程に分散 |
| 日程集中 | 指定した日程に集中投入 |

**パラメータ** (ユーザーが設定):

- 参加可能日程
- 被り上限 (被りなし / n人まで / 制限なし) — 多推し時の同一部での複数メンバー当選制御
- 予算・枚数上限

### 2.3 v3 (コミュニティ・通知)

#### F12: プッシュ通知

- 受付開始・締切のリマインダー
- 支払い期限のリマインダー
- 当日の参加リマインド (「あなたのチケットの第n部があとX分後に始まります」)
- 完売速報 (推しメンの枠が完売したとき)
- 当選した部のカレンダーアプリ同期 (iOS カレンダー / Google カレンダー)

#### F13: コミュニティ完売表

- メンバー全員の完売状況を表形式で表示
- シングルごとの完売推移グラフ
- ユーザー報告数に基づく信頼度スコア

#### F14: 振り返り・メモ

- シングルごとの振り返りメモ
- 「次回への学び」欄

## 3. 非機能要件

### 3.1 パフォーマンス

- 一覧画面の初期表示: 2 秒以内
- マトリックス操作 (タップ→反映): 100ms 以内

### 3.2 セキュリティ

- Supabase RLS により、自分の記録は自分のみ編集可能
- 集計データは読み取り専用で全ユーザーに公開
- 個人を特定できる情報は集計結果に含めない
- **クライアントに含まれる環境変数 (`EXPO_PUBLIC_*`) は公開情報として扱う** — Supabase anon key は RLS で保護されている前提で公開可。service_role key は絶対にクライアントに含めない
- 書き込み系の操作は将来的に Supabase Edge Functions 経由に移行し、サーバー側で検証・監査ログを残すことを検討する
- 重複送信・多重タップ対策として、書き込み API には idempotency key を導入する (v1 実装時)

### 3.2.1 OTA 互換性管理

- EAS Update (OTA) でスキーマ変更を伴う更新を配信する場合、旧クライアントが壊れるリスクがある
- DB スキーマに `schema_version` を持たせ、アプリ起動時に互換性を確認する仕組みを検討する
- `runtimeVersion` の `appVersion` ポリシーで OTA の適用範囲を制御する

### 3.2.2 タイムゾーン

- ミーグリの受付締切・開催日は **JST (Asia/Tokyo)** 基準
- DB には `timestamptz` (UTC) で保存し、表示時に JST に変換する
- 受付締切の境界 (例: 14:00 JST) を扱う際は UTC 変換のズレに注意する

### 3.3 対応プラットフォーム

- iOS 15+
- Android 12+
- (Web は v1 スコープ外、将来対応検討)

### 3.4 多言語対応

- v1: 日本語のみ
- 将来: 英語対応検討

### 3.5 機密情報管理

- **平文の `.env` ファイルは絶対にリポジトリにコミットしない**
- 環境変数の管理には **dotenvx-rs** (Rust 製 dotenvx CLI) を使用する
  - `dotenvx encrypt` で `.env` を暗号化し、暗号化済みファイルをコミットする
  - 復号鍵は `$HOME/.dotenvx/.env.keys.json` にグローバル管理される (プロジェクト内に `.env.keys` を置かない)
  - 開発者が増えた場合、復号鍵 (`DOTENV_PRIVATE_KEY`) は口頭・DM 等でリポジトリ外で共有する
  - 実行時は `dotenvx run -- npx expo start` で復号・注入する
  - 本番環境では `DOTENV_PRIVATE_KEY` を CI/CD のシークレットに設定する
- `.gitignore` に以下を必ず含める:
  - `.env.keys` (万が一ローカルに生成された場合の安全策)
  - `.env` (平文の環境変数)
- **機密情報の漏洩防止は 2 層で対策する**:
  - **ローカル (pre-commit)**: Gitleaks でコミット前に機密情報パターンをスキャン・ブロック
  - **リモート (GitHub Push Protection)**: push 時に既知のシークレットパターンを検出・ブロック (public リポジトリではデフォルト有効)
- 誤検知は `.gitleaksignore` に fingerprint を追加して除外する
- 新規開発者は `make setup` で自動的に pre-commit フックが設定される

### 3.6 インフラ・運用

- バックエンドは **Supabase のみ** で構成 (別途 API サーバーは立てない)
  - Auth / Postgres (RLS) / Edge Functions で全機能をカバー
- 初期は **Free プラン** で運用し、ユーザー定着後に Pro ($25/月) へ移行
- Free プランの自動停止対策 (7 日間無アクセスでプロジェクトが pause される)
  - GitHub Actions 等の cron で 1 日 1 回 Supabase へ軽量クエリを実行し、停止を回避する

### 3.7 環境構成

**dev** と **prd** の 2 環境で運用する。stg は当面不要 (チーム拡大時に必要に応じて追加)。

| 環境 | 用途 | Supabase プロジェクト | 環境変数ファイル |
|------|------|----------------------|-----------------|
| dev | ローカル開発・動作確認 | `supabase start` (ローカル) または Free プランの開発用プロジェクト | `.env` |
| prd | 本番 | Supabase プロジェクト (Free → Pro) | `.env.production` |

- 環境ごとに Supabase プロジェクトを分離し、本番データと開発データを混在させない
- 環境変数ファイルはすべて dotenvx-rs で暗号化管理する
- マイグレーションは `supabase/migrations/` で一元管理し、dev・prd 両方に同じ SQL を適用する

## 4. データモデル (概要)

```
releases  (シングル or アルバム)
  id, title, number, release_date, center_member,
  release_type (single|album)

events  (1リリースに複数紐づく)
  id, release_id,
  format (regular_online|limited_online|real|autograph)

members
  id, name, generation, birthday

event_days
  id, event_id, day_number, date, slot_count

event_day_members  (出演メンバー)
  event_day_id, member_id

reception_rounds  (受付次)
  id, event_id, round_number, start_at, end_at

reception_round_targets  (受付次 × 応募可能な開催日)
  reception_round_id, event_day_id, is_final_round

user_oshi_members  (推しメン)
  user_id, member_id

participation_records  (参加記録)
  id, user_id, event_id, event_day_id, slot_number, member_id,
  status (notApplied|applied|won|lost|cancelled), reception_round_id, memo

purchase_plans  (購買計画)
  id, user_id, event_id

plan_ideal_slots  (理想マトリックス)
  plan_id, event_day_id, slot_number, target_count

plan_actual_slots  (確保状況マトリックス)
  plan_id, event_day_id, slot_number, secured_count

plan_round_slots  (受付次別計画)
  plan_id, reception_round_id, event_day_id, slot_number, planned_count

user_purchase_policies  (購買ポリシー設定)
  user_id, event_id,
  policy (soldout_priority|oshi_maximize|win_rate|date_spread|date_focus),
  available_days (参加可能日程),
  overlap_limit (被り上限),
  budget_limit, sheet_limit (予算・枚数上限)
```

## 5. 画面構成 (v1)
