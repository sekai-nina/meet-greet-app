# Testing Conventions

## フレームワーク

- Jest + React Native Testing Library を使用する

## TDD (バックエンドロジック)

`lib/`、`hooks/`、`stores/` など **UI 以外のロジック層では TDD を徹底する**。

1. **Red** — まず失敗するテストを書く
2. **Green** — テストを通す最小限の実装を書く
3. **Refactor** — テストが通る状態を維持しつつリファクタリングする

TDD の対象:
- `lib/` — Supabase クライアント、ユーティリティ関数
- `hooks/` — カスタムフック (データ取得・加工ロジック)
- `stores/` — Zustand ストア (状態管理ロジック)
- `types/` に付随するバリデーション・変換関数

TDD の対象外 (テストは書くが TDD サイクルは必須としない):
- `app/` — 画面コンポーネント (結合テスト寄り)
- `components/` — UI コンポーネント (視覚確認を優先)

## ファイル配置

- テストファイルは原則、対象ファイルと同階層に `{filename}.test.ts(x)` で配置する
- **ただし `app/` 配下にはテストファイルを置かない** (Expo Router がルートとして認識してしまうため)
- `app/` 配下の画面コンポーネントのテストは `__tests__/app/` に配置する

## 構造

- `describe` ブロックはコンポーネント名または関数名にする
- `it` / `test` の説明は日本語で書く (e.g., `it('ログイン画面が表示される')`)
- AAA パターン (Arrange / Act / Assert) を守る

## モック

- Supabase クライアントはモックする

## アンチパターン

- スナップショットテストは原則使わない (壊れやすいため)
- テストを後回しにしてロジックを先に書かない (TDD 対象の場合)
