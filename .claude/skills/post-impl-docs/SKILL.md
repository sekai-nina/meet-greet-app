# 実装後ドキュメント更新

実装が完了した **後に**、ドキュメントがコードの実態と一致しているか確認・更新する。
実装中に設計変更が入ることは日常的にあり、ドキュメントが古くなりやすい。

## 実行タイミング

- 機能実装が一段落したとき
- PR を作成する前
- スキーマ変更を含む実装が完了したとき

## 手順

### 1. スキーマとデータモデルの整合性チェック

`supabase/migrations/` の SQL ファイルと `docs/requirements.md` のデータモデルセクションを突き合わせる。

- テーブル・カラムの追加/変更/削除が requirements.md に反映されているか
- 型・制約 (NOT NULL, FK, RLS) の記載が実態と一致しているか
- 差分があれば `docs/requirements.md` を更新する

### 2. ユビキタス言語の整合性チェック

実装で使用した型名・変数名が `.claude/rules/ubiquitous-language.md` と一致しているか確認する。

- 実装中に名前を変えた場合、ユビキタス言語を更新する
- 新しい概念を追加した場合、用語を追記する

### 3. ドメイン知識の更新

`docs/domain.md` に実装を通じて判明した新しいドメイン知識があれば追記する。

- 例: 「オフラインミーグリは 1 日 4 部構成」等、実装時に確定した仕様

### 4. ディレクトリ構成の確認

`CLAUDE.md` のディレクトリ構成セクションが実態と一致しているか確認する。

- 新しいディレクトリを追加した場合は反映する

### 5. 画面構成の更新

`docs/requirements.md` の画面構成セクションが実装と一致しているか確認する。

- 画面の追加・変更・削除があれば反映する

### 6. 更新のコミット

ドキュメントに変更があった場合、実装コミットとは **別に** ドキュメント更新としてコミットする。

## 出力

更新内容をユーザーに簡潔に報告する:

- 更新したファイルと変更箇所の一覧
- 差分がなかった場合は「ドキュメントは最新です」と報告する
