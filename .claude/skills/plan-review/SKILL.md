# 設計レビュー (Codex)

Plan モードで作成した設計・実装計画を OpenAI Codex CLI に渡し、第三者視点でレビューしてもらう。
ユーザーが自分では設計の良し悪しを判断しづらい場合に、曖昧な点の洗い出しや代替案の提案を得る。

## 実行タイミング

- Plan モードで計画を作成した後、実装に入る前
- 設計に迷いがあるとき
- ユーザーから「これでいいか確認して」と言われたとき

## 手順

### 1. 計画の整理

現在の Plan の内容を以下の構造でまとめる:

- **目的**: 何を実現しようとしているか
- **技術スタック**: 使用する技術 (CLAUDE.md から引用)
- **設計案**: 具体的な実装方針 (ファイル構成、データフロー、コンポーネント分割など)
- **データモデル**: テーブル設計やスキーマ変更 (該当する場合)
- **懸念点**: 自分で気づいている不安な点

### 2. Codex に送信

計画内容を一時ファイルに書き出し、`codex exec` の引数として渡す。
**注意**: `codex` は stdin からのプロンプト読み取りに対応していないため、必ず引数で渡すこと。

```bash
# 1. 計画内容を一時ファイルに保存
cat <<'PLAN_EOF' > /tmp/plan-review-input.md
{ここに Plan の内容を挿入}
PLAN_EOF

# 2. プロンプトと計画内容を結合して codex exec に渡す
REVIEW_PROMPT="$(cat <<'PROMPT_EOF'
あなたはシニアソフトウェアアーキテクトです。
以下の設計案をレビューし、次の観点で指摘してください。

## レビュー観点
1. **曖昧な点**: 設計として決まっていない・詰めが甘い箇所を質問形式で指摘
2. **設計上の問題**: パフォーマンス、スケーラビリティ、保守性の観点で問題がないか
3. **代替案**: より良いアプローチがあれば提案 (理由付き)
4. **見落とし**: 考慮されていないエッジケースやセキュリティリスク

## プロジェクト概要
- 日向坂46 ミーグリ管理アプリ (React Native + Expo + Supabase)
- 詳細は以下の技術スタック:
  - React Native + Expo (TypeScript strict)
  - Expo Router (ファイルベースルーティング)
  - Supabase (Auth / Postgres / RLS)
  - Zustand + TanStack Query
  - NativeWind
  - dotenvx-rs で環境変数暗号化管理
  - EAS Update (OTA) で配信

## 出力フォーマット
各指摘を以下の形式で出力:
- [質問] — 曖昧な点への質問
- [問題] — 設計上の問題点
- [提案] — 代替案や改善案
- [見落とし] — 考慮漏れ

最後に、この設計案の総合評価を一言で述べてください。

## レビュー対象の設計案
PROMPT_EOF
)"

# 3. プロンプト + 計画内容を結合して実行
codex exec --full-auto -o /tmp/plan-review-result.md "${REVIEW_PROMPT}

$(cat /tmp/plan-review-input.md)"

# 4. 結果を読み取る
cat /tmp/plan-review-result.md
```

### 3. レビュー結果の報告

Codex の出力をユーザーに提示する。構成:

1. **指摘の一覧** — カテゴリ別 (質問/問題/提案/見落とし) に整理
2. **対応案** — 各指摘に対して自分 (Claude) の見解を添える
   - 同意する場合: 計画をどう修正するか提案
   - 同意しない場合: 理由を説明
3. **ユーザーへの確認** — 「以下の修正を計画に反映してよいですか?」と確認

### 4. 計画の更新

ユーザーが承認した修正を Plan に反映する。

## 注意事項

- Codex CLI が利用できない場合 (`codex` コマンドが見つからない場合) はユーザーに通知する
- Codex の指摘は参考意見であり、最終判断はユーザーが行う
- 機密情報 (環境変数の値など) を Codex に送信しない
